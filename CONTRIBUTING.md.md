# 📦 GitHub Workflow (Как работать с репозиторием)

Этот файл описывает, как мы работаем с кодом в команде: как создавать задачи, делать Pull Request и мержить изменения.

---

# 🌿 Ветки (branches)

## Основные ветки
- `main` — стабильная версия проекта (не трогаем напрямую)
- `develop` — основная ветка разработки

## Рабочие ветки
Каждый работает в своей ветке.

Формат названия:
- `feature/название-задачи`
- `fix/название-бага`

Примеры:
- `feature/ui-character-screen`
- `feature/backend-auth`
- `fix/timer-bug`

---

# 🚀 Как работать над задачей

## 1. Забрать актуальный код

Перед началом:
```
git checkout develop
git pull
```

## 2. Создать свою ветку
```
git checkout -b feature/название-задачи
```

## 3. Работать и коммитить

```
git add .
git commit -m "краткое описание изменений"
```

## 4. Запушить ветку
```
git push origin feature/название-задачи
```

---

# 🔀 Pull Request (PR)

После завершения задачи:

1. Заходим в GitHub
2. Нажимаем **Create Pull Request**
3. Выбираем:
   - base: `develop`
   - compare: ваша ветка

## Что писать в PR:
- Что сделано
- Что проверить
- Скриншоты (если есть UI)

---

# 👀 Проверка (Review)

Перед мерджем:
- хотя бы 1 человек смотрит код
- проверяет, что ничего не сломано

---

# 🔁 Merge (мердж)

Если всё ок:

1. Нажимаем **Merge Pull Request**
2. Удаляем ветку

После этого изменения попадают в `develop`

---

# ⚠️ ВАЖНЫЕ ПРАВИЛА

❌ Нельзя:
- пушить напрямую в `main`
- пушить напрямую в `develop`

✅ Нужно:
- всегда работать через ветки
- всегда делать Pull Request

---

# 🔄 Обновление ветки (если develop обновился)

Перед PR желательно подтянуть изменения:

```
git checkout develop
git pull

git checkout ваша-ветка
git merge develop
```

# 🧩 Роли в проекте

## UI/UX дизайнер
- делает макеты в Figma
- при необходимости добавляет ассеты в репозиторий

## Frontend
- реализует интерфейс
- работает с UI и логикой

## Backend
- делает серверную часть
- API и обработку данных
